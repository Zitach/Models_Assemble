Models Assemble 架构计划

一个面向 Claude Code、OpenCode、CC-Switch 等 Coding Agent 的多模型聚合 Provider，把 Kimi、GLM、GPT、DeepSeek 等模型和 coding plan 统一在一个本地或远程网关后面。

目录 项目定位 设计原则 技术栈 目标架构 对外接口 配置设计 安全凭据 Provider Adapter 路由策略 MVP 路线 风险清单 测试策略 推荐第一步

1. 项目定位

Models Assemble 的目标不是再做一个普通的模型 API SDK，而是做一个面向 Coding Agent 的多模型聚合 Provider。

它向 Claude Code、OpenCode、CC-Switch 以及其他支持自定义 Provider 的工具暴露统一模型接口，然后在后端连接 Kimi、GLM、GPT、DeepSeek、OpenRouter、本地模型等不同供应商。

统一入口一个 base_url 接入多个 coding plan 或模型 API。
统一模型名用 assemble-main、assemble-plan、assemble-cheap 屏蔽真实后端。
统一策略集中处理路由、fallback、预算、限速和日志。
Claude Code / OpenCode / CC-Switch

        | Anthropic-compatible / OpenAI-compatible request
        v
Models Assemble Gateway

        v
Kimi Coding Plan / GLM Coding Plan / GPT / DeepSeek / OpenRouter / Local Models

2. 设计原则

不把项目做成绕过限制的脚本

有些 coding plan 可能只允许官方支持的工具或框架接入。Models Assemble 应该定位成 Provider Gateway：客户端仍然是 Claude Code、OpenCode 等 coding agent，网关负责协议适配、路由和可观测性。

协议优先，而不是模型优先

Coding Agent 依赖 streaming、tool calls、tool results、stop reason、usage、long context 和 reasoning 字段。系统核心应该先抽象协议，再接入模型。

Provider 可插拔

Client Request
    -> Normalized Request
    -> Provider Request
    -> Provider Stream / Response
    -> Normalized Event
    -> Client-compatible Stream / Response

3. 推荐技术栈

核心网关推荐使用 Rust。这个项目的主要难点是高并发流式代理、协议转换、长连接稳定性和多 provider 错误恢复，Rust 更适合作为长期核心。

领域推荐用途
HTTP Serveraxum对外暴露 Anthropic/OpenAI-compatible API
异步运行时tokioSSE、上游请求、并发控制
HTTP Clientreqwest调用 Kimi、GLM、GPT、DeepSeek 等 provider
序列化serde / serde_json协议结构转换
日志tracingrequest id、latency、provider 命中记录
CLIclapserve、validate、doctor、test-provider
Python 可以保留给 provider 探测脚本、mock server、回归测试和日志分析。TypeScript / React 可以用于后期 dashboard，但初期不应该先做 UI。

4. 目标架构

MVP 阶段先保持 3 个 crate，等 normalized schema、adapter trait 和路由边界稳定后再拆细。

models-assemble/
  crates/
    ma-cli/             # CLI: serve, validate, doctor, compat-probe
    ma-server/          # HTTP server, auth, SSE, request lifecycle
    ma-core/            # config, protocol, routing, providers, errors
  docs/
  examples/

请求生命周期

  1. Client 发送请求到 /v1/messages 或 /v1/chat/completions。
  2. Gateway 校验本地 API key。
  3. Protocol layer 转换为 NormalizedRequest。
  4. Router 选择 ProviderTarget。
  5. Provider adapter 转换成上游请求。
  6. Gateway 打开上游 streaming 请求。
  7. Provider adapter 把上游 chunk 映射为 NormalizedEvent。
  8. Protocol layer 输出客户端兼容的 stream。

Streaming 与 Tool 状态机

Anthropic:
message_start -> content_block_start -> content_block_delta*
  -> content_block_stop -> message_delta* -> message_stop

OpenAI:
chat.completion.chunk* -> data: [DONE]

内部 stream assembler 要维护 message id、choice index、content block index、tool call id、partial JSON buffer、finish reason、usage 和 upstream state。stream 开始前可以 fallback；已经输出 token 或 tool delta 后不做透明 fallback。

内部错误模型

NormalizedError
  category: auth | rate_limited | timeout | overloaded | invalid_request | provider_bug | network | unknown
  retryable: true | false
  http_status
  provider_code
  safe_message
  raw_debug

5. 对外接口

Anthropic-compatible API

POST /v1/messages
GET  /v1/models

目标是服务 Claude Code 和兼容 Anthropic Messages API 的工具。重点支持 stream、content block events、tool use、stop reason 和 usage。

OpenAI-compatible API

POST /v1/chat/completions
GET  /v1/models

目标是服务 OpenCode、Cline、Roo Code、Continue 等工具。重点支持 SSE、tool_calls、finish_reason、usage 和 model list。

管理 CLI

ma serve --config config.yaml
ma validate --config config.yaml
ma compat-probe --anthropic
ma compat-probe --openai
ma test-provider glm-main
ma list-models
ma doctor

管理 HTTP API 默认不启用。即使启用，也应该默认只绑定 127.0.0.1，并复用本地管理 key。

6. 配置设计

建议使用 YAML。它更适合表达多 provider、多模型别名和路由规则。

server:
  bind: 127.0.0.1:8787
  api_keys:
    - ma-local-dev-key

models:
  assemble-main:
    provider: glm-coding
    model: glm-5.1
  assemble-plan:
    provider: openai-coding
    model: gpt-5.1-codex
  assemble-cheap:
    provider: deepseek
    model: deepseek-v4-flash
  assemble-long:
    provider: kimi
    model: kimi-long-context

providers:
  glm-coding:
    type: zhipu_coding_plan
    base_url: https://example.zhipu.endpoint
    api_key_env: ZHIPU_CODING_API_KEY
  kimi:
    type: openai_compatible
    base_url: https://api.moonshot.cn/v1
    api_key_env: KIMI_API_KEY

routing:
  default: assemble-main
  rules:
    - name: subagent-cheap-model
      when:
        metadata:
          subagent: true
      use: assemble-cheap

fallback:
  assemble-main:
    - assemble-plan
    - assemble-cheap

6.1 安全凭据

默认安全模型是 single-user local gateway。第一版不承诺多用户隔离，也不应该默认暴露到公网。

  • 默认绑定 127.0.0.1，只有用户显式配置时才监听 0.0.0.0。
  • provider key 优先从环境变量、系统 keychain 或 secret file 读取，不建议写入 YAML。
  • ma doctor 检查配置文件权限，发现过宽权限时警告。
  • 日志默认脱敏 Authorization、API key、cookies、token 和常见 secret 字段。
  • 默认不记录 prompt、response、tool 参数正文。
  • provider token 绝不透传给客户端。

缓存边界

第一版不做 response cache。优先做 model list cache、provider health cache、token counting cache 和 provider capability cache。后续 semantic cache 必须支持 TTL、项目隔离、provider 隔离、显式开启和一键清理。

7. Provider Adapter

每个 provider adapter 实现统一接口，隔离供应商协议变化。

trait ProviderAdapter {
    fn name(&self) -> &str;
    fn capabilities(&self) -> ProviderCapabilities;
    fn compliance(&self) -> ProviderCompliance;
    async fn health_check(&self) -> ProviderHealth;
    async fn list_models(&self) -> Vec<ModelInfo>;
    async fn complete(&self, request: NormalizedRequest) -> ProviderResult<NormalizedResponse>;
    async fn stream(&self, request: NormalizedRequest) -> ProviderResult<NormalizedStream>;
}

第一批 adapter 建议从 OpenAI-compatible 开始，然后再做 Anthropic-compatible、GLM Coding Plan、Kimi、DeepSeek、OpenRouter。

能力协商

router 不应该只按模型名路由，还要确认 provider 是否支持 streaming、tools、parallel tool calls、vision、JSON mode、reasoning、max context、max output 和 native protocol。

接入等级

official_api              官方公开 API
official_coding_endpoint  官方文档允许的 coding 工具 endpoint
compatible_proxy          协议兼容，但官方未明确承诺该用途
unsupported               不接入

8. 路由策略

第一阶段：只按请求里的 model alias 路由，例如 assemble-main -> glm-5.1。
第二阶段：支持 fallback，例如 timeout、429、5xx 时切到备用模型。
第三阶段：支持规则路由，例如长上下文走 Kimi，子任务走 cheap model，规划任务走强 reasoning 模型。

早期只支持显式 model alias、上下文长度、provider health、错误 fallback。任务语义分类放到后期。

限流与预算

第一版建议支持 local client rate limit、per-provider concurrency limit、per-provider retry budget、daily/monthly token cap、daily/monthly cost cap 和 fallback cost policy。默认不应在用户无感知的情况下从便宜模型 fallback 到昂贵模型。

9. MVP 路线

Milestone 0：Rust workspace、CLI、配置加载、/health，以及 ma compat-probe 最小 mock provider，用来验证 Claude Code / OpenCode / CC-Switch 是否接受自定义 base_url、模型列表和 SSE。
Milestone 1：OpenAI-compatible /v1/chat/completions，支持 non-stream、stream、tool_calls、错误映射和 client disconnect 取消上游请求。
Milestone 2：模型别名、fallback、命中日志。
Milestone 3：Anthropic-compatible /v1/messages，支持 Anthropic SSE event。
Milestone 4：接入 GLM、Kimi、GPT coding plan，建立能力矩阵。
Milestone 5：ma doctor、usage、latency、fallback chain、dashboard 可选；默认不采集 prompt、response 和 tool 参数正文。

10. 风险清单

风险说明处理策略
Claude Code 兼容性需要验证自定义 base_url、SSE、tool use 和错误格式。优先做真实客户端集成测试。
Coding Plan 限制某些套餐可能限制接入方式。只支持官方明确允许的方式，实验方案明确标记。
Tool Use 差异OpenAI 和 Anthropic 的 tool call streaming 语义不同。用 normalized event 做隔离，并针对多 tool call 测试。
Streaming 边界上游超时、下游断开、半截 JSON、请求取消。stream 开始前允许 fallback，stream 开始后不做透明 fallback。

11. 测试策略

单元测试

覆盖协议转换、provider event 转换、routing rule 和 fallback decision。

集成测试

使用 mock provider 覆盖正常 streaming、429、500、timeout、malformed chunk 和 tool call chunk。

真实客户端测试

OpenCode -> Models Assemble -> mock/OpenAI-compatible provider
Claude Code -> Models Assemble -> mock/OpenAI-compatible provider
CC-Switch -> Models Assemble -> provider

12. 推荐第一步

第一步不要直接接 GLM、Kimi、GPT 三家。先证明网关闭环。

OpenCode
  -> Models Assemble /v1/chat/completions
  -> OpenAI-compatible adapter
  -> 任意普通 OpenAI-compatible 模型

确认 OpenCode 能把你的服务当 provider、SSE streaming 没问题、模型别名能工作、日志能看到完整链路。之后再做 Anthropic-compatible 入口，最后接入具体 coding plan。

最终边界：Models Assemble 不替代 Claude Code / OpenCode，而是成为它们背后的多模型 Provider。
